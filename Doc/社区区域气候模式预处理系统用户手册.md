## 1. 引言

社区区域气候模式预处理系统（CRESM Preprocessing System，CPS）是为新一代国产区域地球系统模式 CRESM 开发的数据前处理系统，用于统一管理和自动化执行模式运行前所需的各类数据准备任务。CRESM 基于区域气候模式 CWRF，并进一步耦合了新一代通用陆面模式 CoLM2024，通过耦合器架构实现陆—气过程的协同模拟，对数据一致性和前处理流程的规范性提出了更高要求。

在实际应用中，CRESM 的前处理工作涉及多个独立子系统和复杂的数据依赖关系，包括 CWRF 所需的静态地理数据与侧边界驱动场准备、CoLM 所需的基础地表参数场与初始状态场构建，以及 CPL7 耦合器所需的网格映射与插值权重文件生成等步骤。这些流程通常分散在不同工具和脚本中，配置复杂、操作繁琐，且缺乏统一的流程管理机制，容易产生人为配置错误和重复劳动。

CPS 的目标是为 CRESM 提供一套统一、自动化、高性能的数据前处理解决方案。系统通过模块化设计对各前处理环节进行标准化封装，并提供统一的配置接口和流程控制机制，使用户能够以较少的人工干预完成复杂的前处理任务。同时，CPS 支持多平台环境部署，可适配不同计算节点与运行环境，为 CRESM 的批量实验和长期业务化运行提供稳定可靠的技术支撑。

## 2. 系统环境要求

CPS 系统面向高性能科学计算环境设计，主要运行于 Linux 平台，用于支持 CRESM 模式体系的数据前处理工作。在安装和使用 CPS 之前，请确保计算环境满足以下基本要求。

---

### 2.1 操作系统要求

推荐使用基于 Linux 的操作系统环境：
- Linux
	- Ubuntu 20.04 及以上版本(推荐)
- 或其他支持运行CRESM的操作系统

---

### 2.2 硬件配置要求

推荐配置（具体需求取决于区域大小和数据规模）：
- CPU：48核及以上
- 内存：100 GB RAM
- 磁盘空间：至少 100 GB 可用空间

---

### 2.3 系统软件依赖

CPS 运行过程中依赖多个底层科学计算库和并行计算组件，请确保以下软件已正确安装：
- Intel Fortran 编译器套件
- Intel MPI 并行环境
- NetCDF-C 与 NetCDF-Fortran 库 
- HDF5 数据库库
- netCDF Operator (NCO)
- Climate Data Operators (CDO)
- NCAR Command Language (NCL)

建议用户在系统环境中验证以下工具可正常使用：
- `mpirun`
- `ncdump`
- `ncl`

---

### 2.4 Python 运行环境

CPS 基于 Python 运行，建议使用 Conda 进行环境管理，具体环境配置请参考4.1节。
Python 版本要求：
- Python 3.9 及以上版本

主要依赖库包括：
- numpy
- pandas
- matplotlib
- cartopy
- Pillow
- geopandas
- fiona
- esmpy
- esmf
- xesmf
- pyhdf
- pyparsing
- python-dateutil

---

### 2.5 模式系统依赖

在运行 CPS 之前，需要提前完成以下模式组件的安装与编译配置：
- 区域气候模式 CRESM
- 区域气候模式CWRF的前处理套件(CWPS)

上述组件必须能够独立正常运行，以保证 CPS 前处理流程的正确执行。
我们提供一个标准的工具库CRESM_ToolBox去完成上述系统依赖安装。

### 2.6 运行环境配置要求

推荐使用单独的环境配置文件（例如: `~/.bashrc_cresm`）管理CPS运行环境。

## 3. 目录结构

CPS 采用模块化目录结构设计，以保证数据、配置文件和程序代码之间的清晰分离，便于系统维护、批量实验管理和跨平台部署。本节对 CPS 的主要目录结构及其功能进行说明。

---

### 3.1 项目根目录结构

CPS 项目目录结构如下：

```bash

CPS/
	├── PrepScript/          # 核心前处理脚本
	├── SpinUpScript/        # CoLM Spin-Up脚本
	├── CRESM_ToolBox/       # CRESM 工具箱
	├── Case/                # 个例与实验工作目录
	└── README.md            # 使用说明文档

```

---

### 3.2 核心程序目录（PrepScript）

```bash

PrepScript/
	├── CRESM_Preprocessing_System.py   # 主入口程序
	├── PrepCWRF.py                     # CWRF 前处理模块
	├── PrepCoLM.py                     # CoLM 前处理模块
	├── PrepCRESM.py                    # CPL7 映射文件生成模块
	├── env.ini                         # 环境配置文件
	├── case.ini                        # 实验配置文件
	├── Utils/                          # 内部工具库
	├── ProcessScript/                  # 外部处理脚本库
	├── NML/                            # 模型namelist模板库
	└── Forcing/                        # 驱动数据信息库
```

功能说明：
- `CRESM_Preprocessing_System.py`
	系统主入口脚本，负责流程调度与模块调用。
	
- `PrepCWRF.py`
	用于执行 CWRF 静态数据处理、IC/BC 生成等流程。
	
- `PrepCoLM.py`
	负责陆面模式 CoLM 的基础数据准备与初始场构建。

- `PrepCRESM.py`
	用于生成耦合器所需的网格映射和插值权重文件。

---

### 3.3 实验工作目录（Case）

```bash

Case/
├──[GridName]/                    # 实验数据汇总目录
	├──Create_Run_From_CPS.py     # 用以创建运行案例的脚本
	├──Grid_[GridName]/           # 静态数据
	└──ICBC_[GridName]/           # CWRF初始场及驱动场数据
├──PrepCWRF/                      # CWRF相关数据制作目录
	├──[GridName]/                # CWRF相关数据汇总目录
	├──First_StaticData/          # 静态数据制作目录
		└──*Geog_[GridName]/      # CWRF geog数据目录 (对同一区域可复用)
	└──Second_ICBC/               # CWRF初始场及驱动场数据制作目录
├──PrepCoLM/                      # CoLM相关数据制作目录
	├──[GridName]/                # CoLM相关数据汇总目录
	├──First_GenMesh/             # CoLM与CWRF网格映射文件制作目录
	├──Second_MakeSrf/            # CoLM相关静态数据制作目录
		└──*CoLMSrf_[GridName]    # CWRF geog数据目录 (对同一区域和define配置可复用)
	└──Third_Remap/               # CoLM数据remap至CWRF网格
├──PrepCRESM/                     # CRESM相关数据制作目录
	└──[GridName]/                # cpl7相关数据汇总目录
├──NMLS/                          # namelist
└──Log/                           # 日志文件目录
```

说明：
- 每个 Case 目录对应一个独立实验区域或分辨率配置
- `Geog_[GridName]` 和`CoLMSrf_[GridName]`分别为CWRF和CoLM的静态数据目录，在同一实验配置下可复用。 实验配置由唯一的Geog.ID验证。
- 系统会在该目录下自动生成各模块运行所需的中间文件和结果数据。
- 用户通常只需要管理 Case 层级目录，无需修改内部自动生成文件。
- **警告:** 如果在实验配置文件中启用了自动清理缓存(CleanTempFiles)， 那只有 `Case/[GridName]/` 文件夹会被保留，请谨慎使用。

## 4. CPS流程总览

CPS提供了一个集成且自动化的工作流程，该工作流程旨在以统一且可复现的方式处理大气强迫、陆面数据集和耦合器映射文件的预处理。CPS 通过用户定义的配置文件控制的分阶段管道执行预处理任务，总体执行流程如下

```bash

		case.ini + env.ini                # 用户配置文件
				 ↓
	┌──────────────────────────┐
	│  CRESM Preprocessing     │          # 主程序
	│        System            │
	└──────────────────────────┘
				 ↓
	┌──────────────────────────┐
	│  PrepCWRF                │          # 准备CWRF所需数据
	│  - Domain Visualization  │          # 可视化研究域
	│  - Static Geogrid        │          # geogrid
	│  - Vegetation Parameters │          # 植被参数(FVC, LAI, SAI) 
	│  - Initial/Boundary Data │          # 初始场/测边界驱动场
	└──────────────────────────┘
				 ↓
	┌──────────────────────────┐
	│  PrepCoLM                │          # 准备CoLM所需数据
	│  - Mesh Generation       │          # 制作映射文件
	│  - Surface Data          │          # 准备CoLM的陆面数据
	│  - Initial Conditions    │          # 初始化CoLM
	│  - Spin-up & Remapping   │          # Spin-up和重网格
	└──────────────────────────┘
				 ↓
	┌──────────────────────────┐
	│  PrepCRESM               │          # 准备CRESM所需数据
	│  - Coupler Mapping Files │          # 耦合器权重文件
	└──────────────────────────┘
```

## 5. 安装与部署

### 5.1 Python 环境部署

CPS 的 Python 运行环境依赖多个科学计算、地理信息处理和数据可视化相关库。为保证依赖版本的一致性与跨平台兼容性，所有 Python 依赖已统一维护在CRESM_ToolBox目录的三个环境配置文件中。为防止环境冲突，我们需要新建3个python环境。

*请注意在现有版本中，用户只能通过conda进行环境配置。*

用户无需手动逐一安装各个依赖库，只需通过以下方式完成环境配置：

```bash

conda env create -f cresm.yml              

conda env create -f cresm_xesmf.yml

conda env create -f Chaomodis.yml

```

以上命令会新建三个新环境：
- `cresm`: 负责运行`main`, `FlowDir`, `plot`等脚本
- `cresm_xesmf`: 负责运行`GenMesh`, `Remap`等脚本
- `Chaomodis`: 负责运行`FVC`, `IGBP`, `LAI/SAI`

---

### 5.2 CRESM 工具箱安装

CPS提供了一个完整的CRESM工具箱，该工具箱包含区域气候模式CWRF的前处理系统CWPS和依赖数据集，CWRF陆面数据制作相关代码及依赖数据。我们为用户提供了便捷的安装脚本`SetupAll.py`. 用户可以在设置完`Makeoptions.ini`之后通过`SetupAll.py`一次安装部署所有工具。`Makeoptions.ini`包含：

```ini

[LIBPATHS]
NETCDF_path = $software/netcdf-single-r        # netcdf 安装目录   
JASPER_path = $software/jasper                 # jasper 安装目录
PNG_path    = $software/libpng                 # libpng 安装目录 
ZLIB_path   = $software/zlib-1.2.11            # zlib 安装目录
```

配置完上述路径后，您可以通过以下命令完成安装：

```bash
python SetupAll.py
```

# 6. 配置文件

---

### 6.2 实验配置文件 （`case.ini`）

case.ini 用于控制 “做什么 + 在哪里 + 做多久”，决定 CPS 的执行流程与目标实验。

#### 6.2.1 **[BaseInfo]** -- Global Workflow Settings

| 参数名              | 类型   | 说明                                           | 示例         |
| ---------------- | ---- | -------------------------------------------- | ---------- |
| CleanTempFiles   | bool | 若为True，则删除中间文件以节省磁盘空间。                       | True/False |
| Use_CoLMLAI      | bool | 若为True，则使用 CoLM 的 LAI 数据而不是默认的 MODIS 数据制作LAI | True/False |
| Enable_TimeChunk | bool | 启用时间分割功能以对制作ICBC是进行并行（仅在ICBC是有用）             | True/False |
| TimeChunkCount   | int  | 将运行过程分割成的块数（建议 4-10 个，请考虑IO瓶颈）               | True/False |

| MeshSize        | int   | CoLM与CWRF的非结构化网格映射文件分辨率<br>1=Coarse (21600x43200), 2=(43200x86400) 3=Fine (86400x172800)<br>*3km及以下分辨率建议选择2， 3km以上分辨率选择1* | 1                   |

> [!note]
> 用户可以同时定义多个实验配置（例如，[CN_15km]、[Yangtze]）

## 7. 基本用法

### 7.1 使用前准备

在运行 CRESM 数据前处理系统之前，请确保以下准备工作已完成：
1. 系统环境已正确配置
	- 已完成 env.ini 中所有路径与环境的配置；
	- 相关系统环境（CWRF、CoLM）与 Conda 环境均可正常激活；
	- 所有外部程序（如 ncl、cdo、ncks 等）可在命令行直接调用。

2. 实验配置文件已准备
	- 针对具体实验案例，正确填写 case.ini；
	- 确保 ForcingDataName 与 env.ini 中对应的 forcing 配置一致；
	- 检查模拟时间、网格参数、分辨率等关键信息。

### 7.1 命令行参数说明

CRESM 数据前处理系统通过统一的主程序启动，提供了诸多选项，用户可以是同`-h`选项查看命令行帮助

python CRESM_Preprocessing_System.py -h

---

#### 7.1.1 `-v, --version`

**显示程序版本信息并退出。**
- 用于确认当前使用的 CPS 版本
- 便于问题反馈与版本对齐

---

#### 7.1.2 `-d, --debug`
**启用调试模式（Debug Mode）。**
- 打印更详细的日志信息
- 包括路径检查、环境检查、每一步调用的外部命令
- 建议在出现错误时开启

---

#### 7.1.3 `-ch, --confighelp`
**显示配置文件说明并退出。**
- 打印 `case.ini` 和 `env.ini` 的完整配置说明
- 用于快速查阅配置含义，无需打开手册

---

#### 7.1.4 `-l, --listcases`
**列出 case.ini 中已定义的可用实验案例并退出。**
- 自动扫描 case.ini
- 显示所有可直接运行的 [GridName] 小节
- 不执行任何计算流程

---

#### 7.1.5 `-n NAME, --gridname NAME`
**指定要运行的实验案例名称。**
- NAME 必须与 case.ini 中的某个 [GridName] 小节一致
- 这是实际运行 CPS 时的核心参数
- 
---

#### 7.1.6 `-g DIR, --geogdir DIR`
**使用已有的 CWRF 地理静态数据目录。**
若指定该参数：
- 跳过 `Geogrid/FVC/LAI/SAI` 等静态地理数据生成步骤
- 直接使用 DIR 中已有的数据
- 网格配置有`Geog_[GridName]`中的`Geog.ID`文件确认。不同网格配置不能混用。

Geog_ID样式：
```bash
X[232]_Y[172]_dx[30000.000000]_dy[30000.000000]_lat[35.177800]_lon[110.000000]_
tl1[30.000000]_tl2[60.000000]_slon[110.000000]_bw[15]_lk[0.500000]
```

---

#### 7.1.7 `-s DIR, --colmsrf DIR`
**使用已有的 CoLM 地表数据目录。**
若指定该参数：
- 跳过 `PrepCoLM` 中的地表数据生成流程`MakeSrf`
- 网格配置有`CoLMSrf_[GridName]`中的`Geog.ID`文件确认。不同网格配置不能混用。
- 多案例共享同一套 CoLM surface 数据建议使用

---

#### 7.1.8 `-y YEAR, --year YEAR`
**覆盖`case.ini`中定义的起止年份，仅运行指定年份， 这会自动生成新实验`[GridName].year`。**
- YEAR 为四位整数（如 2023）
- 自动调整：
	- StartTime (如：2023-12-01_00:00:00)
	- EndTime (如：2025-01-05_00:00:00)
	- 起止时间为`year-1年12月1日`至`year+1年1月5日`
- 主要用于逐年强迫数据处理, 或跨年运行。

---

#### 7.1.9 `-c NAME, --collectcase NAME`
**年度模式下的数据汇总参数（与 `--year` 联合使用）。**
- 将多个年度的强迫数据汇总到同一目录 NAME
- 每个文件自动追加年份后缀
- 便于后续多年模拟或批处理
**注意：仅在使用 `--year` 后单独使用才有效**

### 7.2 运行

#### 7.2.1 最小运行示例

当第一次运行一个新案例时推荐使用以下指令运行：
```bash
python CRESM_Preprocessing_System.py -n CN_30km
```

---

#### 7.2.2 制作多年数据
当需要制作多年数据时，推荐先只运行5天，生成静态数据`CoLMSrf_[GridName]` 和 `Geog_[GridName]`。
```bash
python CRESM_Preprocessing_System.py -n CN_30km
```

之后复用静态数据再生成多年数据 (以2000年为例)
```bash
python CRESM_Preprocessing_System.py -n CN_30km -y 2000 \
									 -g path/to/Geog_[GridName] \
									 -s path/to/CoLMSrf_[GridName]
```

再所有年份都运行完毕后，在使用收集模式对所有年度数据进行收集
```bash
python CRESM_Preprocessing_System.py -c CN_30km
```

---

### 7.2.3 调试运行

由于CPS是一个十分复杂的系统，因此当出错时可能需要使用调试模式以确定出错的具体环节。建议当出错时仅开启当前流程(`Go_`) ，并启用`-d`模式
```bash
python CRESM_Preprocessing_System.py -n CN_30km -d
```

## 8. 创建运行案例
通过CPS制作的数据既可以使用以前的`create_run.py`创建运行案例，也可以使用更方便的`Create_Run_From_CPS.py`创建运行案例。`Create_Run_From_CPS.py`有以下可选参数：

```bash
Create run directory for CRESM

optional arguments:
  -h, --help            show this help message and exit
  -casename CASENAME, --casename CASENAME
                        name for entire run
  -casedir CASEDIR, --casedir CASEDIR
                        The parent directory where the case will be created
  -griddir GRIDDIR, --griddir GRIDDIR
                        Full path to Grid directory
  -icbcdir ICBCDIR, --icbcdir ICBCDIR
                        Full path to ICBC directory
  -st STARTTIME, --starttime STARTTIME
                        YYYYMMDDHH
  -et ENDTIME, --endtime ENDTIME
                        YYYYMMDDHH
  -ncpus NCPUS, --ncpus NCPUS
                        number of cpus
  -jms {lsf,slurm}, --job_management_system {lsf,slurm}
  -ini INI, --ini INI   config.ini path
  -yearly, --yearly     If true, link ICBC files without renaming 
						and copy wrfinput for start year
  -fvcom FVCOM, --fvcom FVCOM
                        True/False
```
其中，`casename`是案例名称，`casedir`是案例创建目录，`griddir`是静态数据目录(即`Grid_[GridName]`)。`icbcdir`是CWRF初始化和测边界场数据。`starttime`和`endtime`分别是开始时间和结束时间。`ncpus`是使用的核心数。`job_management_system`是任务管理系统，目前仅支持`lsf`, `slurm`, `ini` 是prep.ini文件的路径。`yearly`用以区分是否是分年的ICBC。如果数据是分年(即有年份后缀)，但是未启用，则只链接第一年的数据。

## 9. 日志
CRESM Preprocessing System（CPS）采用分级、分流程的日志机制，以便用户在复杂的数据前处理过程中进行状态跟踪、问题诊断和结果回溯。
日志系统主要由**主日志（Main Log）和流程日志（Process Logs）**两部分组成。

---

### 9.1 主日志（Main Log）
**日志位置**
主日志文件默认保存在当前运行目录，文件命名格式为：
```bash

DataPrepare.[GridName].log

```
其中 `GridName` 对应命令行参数 `--gridname` 指定的案例名称。
**主日志用于记录整个 CPS 工作流的全局信息，主要包括：**
-  程序启动与结束信息
- 配置文件读取与参数解析结果
- 系统环境与路径检查结果
- 各预处理模块的开始与结束状态
- 外部程序调用命令（Debug 模式下）
- 全局错误与异常回溯信息
**日志级别:** 
- `INFO`：关键流程节点与总体状态信息
- `DEBUG`：详细执行信息（仅在启用 --debug 时输出）
- `WARNING`：非致命问题提示
- `ERROR`：流程中断或执行失败信息

**建议：**
- 日常生产运行可关闭 Debug
- 初次部署或出错时开启 `--debug`

---

### 9.2 流程日志（Process Logs）
**日志位置**
主日志文件默认保存在当前运行目录，文件命名格式为：
```bash
CaseOutputPath/[GridName]/Log/
```
该目录用于集中存放各个处理步骤的标准输出与错误输出。

**命名规则**
流程日志采用如下统一命名格式：
```bash
log.<ProcessName>[.<TimeTag>]
```
在涉及时间分片或多次调用的流程中，日志文件可能附带时间标识或编号后缀。

**日志内容**
流程日志主要记录：
- 外部可执行程序（如 `geogrid.exe`, `ungrib.exe`）的标准输出
- MPI 并行运行信息
- 模型或工具自身产生的错误信息
- 详细的数值或文件处理过程
这些日志通常是定位失败原因的第一手信息。

### 9.3 日志与错误定位建议
当 CPS 执行失败时，建议按以下顺序进行排查：
1. 查看主日志
	- 确认失败发生的流程阶段及路径 （会提示`Error at:`）
	- 检查环境、路径或参数相关错误

2. 查看对应流程日志
	- 打开 `CaseOutputPath/[GridName]/Log/log.<ProcessName>`
	- 检查是否存在：
		- 库缺失（如 libnetcdf*.so）
		- 输入数据不存在
		- MPI 或并行错误

3. 开启 Debug 模式重试
> [!danger]
> 若启用了 CleanTempFiles = True，部分中间文件可能被自动删除，因此Debug时请勿开启该模式

### 9.4 常见错误排查建议

在目前版本中，部分操作系统可能会出现以下错误：
**1. 无法找到依赖库，如netcdf**
- 请确保在运行CPS前，source对应环境后重试，目前在CPS内部切换环境的功能并不稳定

**2. CWRF制作ICBC数据时出错且原因未知**
- 部分网格的划分数会与分配的核心数冲突，因此可以尝试更改分配的核心数

**3. CoLM部分允许失败且原因未知**
- 可能时由于环境加载有误，请重新加载环境再次允许

> [!caution]
> 请注意，CoLM的define选项会影响你的目标实验配置，请确保define正确
